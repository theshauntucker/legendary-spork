## What shipped this session

### Infrastructure: GitHub Actions migration runner
The sandbox running these Claude Code sessions cannot reach `*.supabase.co` or `api.supabase.com` (network allowlist is GitHub-only), so a CI-side runner is the long-term home for database migrations.

- `.github/workflows/apply-migrations.yml` — triggers on `workflow_dispatch` AND on pushes that touch `supabase-coda-*.sql`, `supabase/migrations/**.sql`, or `scripts/apply-*.mjs`. Pinned to project ref `xkckvrbxaessudolxhte`.
- `scripts/_supabase-api.mjs` — thin wrapper around Supabase Management API's `POST /v1/projects/{ref}/database/query`. Reads `.env.local` for local dev.
- `scripts/apply-migration.mjs` — one-shot migration applier: `node scripts/apply-migration.mjs path/to/file.sql`.
- `scripts/apply-all-pending.mjs` — scans repo root for `supabase-coda-NNN-*.sql`, ensures a `public._applied_migrations` tracker table, applies anything not yet in the tracker, uploads the resulting `apply-migrations.log` as a workflow artifact in JSON-lines format.

After the secret is added, **every future push that adds a `supabase-coda-*.sql` auto-applies on merge** — no more manual clicking.

### P-PRE-1 — Error-status sweeper bug fix
- **Bug:** The old process route's outer try/catch marked `videos.status = 'error'` on ANY exception, including downstream bookkeeping (email send, credit deduction) that had nothing to do with the analysis succeeding. Users saw "analysis failed" screens for routines that actually had valid scores.
- **Fix:** `src/app/api/process/route.ts` now flips `videos.status = 'analyzed'` as the **first** DB write after Claude Vision returns a valid score. Analysis insert, `analysis_id` linking, credit deduction, and admin notification are each wrapped in their own try/catch. The outer error-handler re-reads the current status and only flips to `error` if the row isn't already `analyzed`.
- **Backfill:** `supabase-coda-012-backfill-error-status.sql` recovers any video row stuck at `status='error'` that has a matching `analyses.total_score` (> 0), and links `analysis_id` where missing.

### P-PRE-2 — Perceptual dHash duplicate detection
- `src/lib/dhash.ts` — 64-bit dHash (9×8 grayscale diff) + Hamming distance. Cross-browser (no BigInt; four 16-bit XORs + Kernighan popcount).
- Client-side (`src/lib/video-processor.ts`) computes a dHash for each of the first 3 frames during extraction (reusing the canvas already in memory).
- `src/app/upload/page.tsx` forwards the `phashes` array to `/api/analyze`.
- Server (`src/app/api/analyze/route.ts`) stores them in new `videos.frame_phash` column and runs a fuzzy cross-user lookup over the latest 2000 rows. Threshold: min Hamming distance ≤ 8 bits. Matches return HTTP 409 with `code: DUPLICATE_VIDEO`; the message differs for same-owner vs cross-user so legit re-uploads still get a helpful path.
- `supabase-coda-013-frame-phash.sql` adds `frame_phash text[]` + partial index.
- The existing exact-match `frame_fingerprint` column is **kept** during rollout as the fast path, per rollout plan.

### P1 — Web Push + in-app notifications
- VAPID keypair generated. Public key committed; private key in env var handoff above.
- `supabase-coda-014-push-subscriptions.sql` adds `push_subscriptions` (one row per browser install, unique endpoint) and `notifications` (bell-icon feed). RLS locks both to owner-read, service-role-insert.
- `src/lib/push.ts` (server) — web-push dependency, VAPID config with graceful no-op when key missing, auto-detects dead endpoints for pruning.
- `src/lib/push-client.ts` (browser) — SW registration, permission, subscribe/unsubscribe flow.
- `public/sw.js` — dependency-free service worker that handles `push` + `notificationclick`.
- `src/lib/notify-user.ts` — one-call fan-out: writes `notifications` row, pushes to every active subscription, prunes 404/410 endpoints.
- `src/app/api/push/vapid/route.ts` GET — returns the public key the client needs.
- `src/app/api/push/subscribe/route.ts` POST/DELETE — upsert / remove a subscription.
- `src/app/api/notifications/route.ts` GET (list + unread count) / POST (mark read / mark all).
- `src/components/NotificationBell.tsx` — glass popover in the Navbar with unread badge, ask-to-enable-push prompt, and deep-links.

### Notification fan-out wired into existing features
- **Follow** (`src/app/api/follow/route.ts`): new follower → "X followed you" deep-links to their profile.
- **Comments** (`src/app/api/comments/route.ts`): top-level comment → post owner. Reply → the parent comment's author (not the post owner). Never self-pings.
- **Reactions** (`src/app/api/reactions/route.ts`): reaction on a post → owner with emoji glyph in the headline.
- `src/lib/post-owner.ts` — shared resolver mapping `(post_type, post_id)` → author `user_id` so fan-out always hits the right inbox.

### Moderation primitives
- `supabase-coda-015-moderation.sql` creates `public.reports` and `public.blocks` with full RLS + a `public.is_blocked(viewer, target)` helper for symmetric mute (feed + DM can call it).
- `src/app/api/moderation/report/route.ts` POST — records a report with idempotent upsert on `(reporter, target_kind, target_id)`. Admin email via new `notifyAdminReport` in `src/lib/notifications.ts`.
- `src/app/api/moderation/block/route.ts` POST/DELETE/GET — manage blocks; on block, also clears mutual follows so the UX takes effect immediately.

### Discovery indexes
- `supabase-coda-016-search-indexes.sql` enables `pg_trgm` and adds GIN trigram indexes on `profiles.handle`, `profiles.display_name`, `studios.name`, `studios.city`, `choreographers.name`. The `/find` page's `ilike '%q%'` queries scale past seq scans.

## Known gaps (not launch-blockers)

These exist as APIs without user-facing UI because the launch focus is the feed + analyzer + DM. Post-launch polish:

- **Moderation UI.** `/api/moderation/report` and `/api/moderation/block` work; there's no "Report / Block" menu in FeedCard / ConversationView yet. An admin console for reviewing `public.reports` is also pending.
- **Feed-filter for blocks.** Blocks work in the database (RLS + `is_blocked()` helper) but the feed-composition query in `src/app/home/*` doesn't yet filter by `is_blocked()`. Low-priority until we actually see reports come in.
- **Bond / Check-in / Thread notifications.** Only follow/comment/reaction are wired. Plumbing is trivial now that `notifyUser` exists; not done here.
- **Studio dashboard deep notifications.** Studio owners don't get pinged on claim-accepts, roster changes, etc. Again: `notifyUser` helper is ready.
- **`ALL_EVENTS` is static.** `/find` events tab reads from the baked-in dataset at `src/data/competitions.ts`. Moving to DB-backed is a post-launch lift.
- **Lint tooling.** `next lint` is deprecated and no ESLint config exists — so `pnpm lint` fails. `pnpm build` (with full type-check) is the source of truth for now.

## Known issues

- **Dead VAPID subscribers** prune lazily — if a browser unregisters SW without calling our DELETE endpoint, the first push after that will hit 404/410 and be cleaned up on that send. No periodic cron needed.
- **Cross-user dHash threshold.** 8-bit Hamming distance is the spec; in testing that catches re-encodes, trims, and watermark additions. It might also false-positive on solo-only close-up re-shots of the same costume against the same backdrop. If reports come in, bump to 6.
- **The migration workflow's first push WILL fail red** until `SUPABASE_ACCESS_TOKEN` is set on the repo. That's expected. Add the secret, click Run workflow, it goes green.
